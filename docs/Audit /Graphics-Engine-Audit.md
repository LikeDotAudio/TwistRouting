# Audit — The Graphics Engine

**Companion to:** `Flexible-Software-Media-Production-Requirements-Audit.md` (App IDs `AR.x`), `TwistRouting-vs-Requirements-Delta.md` (L0–L4 scale), `Signal-Conditioner-Row-Design-Audit.md`.

**What this audits.** How to build a broadcast **graphics engine** — the thing that creates on-air titles, sub-titles, overlays, animations, fly-ins, transitions, alpha layers, presets, "coming-up" lists, participant/credit lists — and *how it all actually works* end to end. Grounded in real-world engines (CasparCG, Vizrt, Ross XPression, Chyron, Singular.live, Viz Flowics) and mapped to the requirements catalog and the TwistRouting codebase. Client identifiers stripped; technology only.

---

## 0. The Critical Framing — Renderer vs. Engine

The requirements catalog contains an **HTML5 Graphics Renderer App (AR.4.7)** — but read it carefully and it is deliberately *dumb*:

> "…a simple app that leverages standard HTML5 graphics… The HTML5 pages are generated by an **upstream graphics workflow**, which produces graphics templates and populates them with various data feeds and user input. **The upstream graphics workflow is out of scope** for this document. Instead this App is simply concerned in rendering the HTML5 pages…"

So the request specs only the **renderer**: point a headless Chromium at a URL, render "blindly," and emit **Fill + Key (alpha)** onto the data plane (as `4:2:2 + 4:0:0` YUV fill+key **or** RGB+Alpha), optionally with up to 16 channels of associated audio. It has "no controls or triggers to start or stop graphics" — unless it integrates with the upstream tool.

**That upstream tool — the authoring/template/data/animation/playout system — is the "graphics engine," and it is exactly what this audit covers.** It is the part the request left out of scope, and the part with real creative and technical depth. A renderer that "blindly" paints a web page is trivial; the engine that *produces and drives* that page is the product.

```
        ┌──────────────────────────────────────────────────┐
        │              GRAPHICS ENGINE (this audit)         │
        │  Design/Template → Data → Control/Playout         │
        └───────────────────────┬──────────────────────────┘
                                 │ HTML5 page URL + trigger/data commands
                                 ▼
                 AR.4.7 HTML5 RENDERER  (headless Chromium)
                                 │ Fill + Key (alpha) onto data plane
                                 ▼
                 Downstream Keyer (DSK) / switcher  →  PROGRAM
```

## 1. The Four-Layer Architecture (how every real engine is built)

Every production graphics system — from open-source CasparCG to Vizrt to cloud Singular.live — is the same four decoupled layers. This is the reference architecture to build against.

| Layer | Responsibility | Real-world examples |
|---|---|---|
| **1. Design / Template** | A designer authors a reusable *template* = a scene/layer tree + animations + a **schema of named, empty data fields**. Design is separated from data so one template serves thousands of on-air instances. | Vizrt **Viz Artist** scene tree of *Containers*; CasparCG template = an **HTML page**; Singular **Composer** compositions with exposed control fields; Ross XPression scenes / "super-templates". |
| **2. Data** | Live/operator values bound into the template's fields at runtime, from JSON/REST, **WebSocket** streams, spreadsheets, databases, or sports/election feeds. | Ross **DataLinq**; Vizrt **DataPool/DataCenter**; Singular **Data Streams / JS Data Nodes**; CasparCG `CG UPDATE {json}`. |
| **3. Control / Playout** | An operator (or automation, or a newsroom rundown) decides *which* template, *when*, and issues **take / next / update / out** — with a **preview** vs **program** distinction. | Rundowns/playlists (Viz Trio, XPression sequencer, WASP3D); CasparCG **AMCP**; newsroom **NRCS + MOS** protocol. |
| **4. Render + Output** | GPU renders frames at the channel frame rate, **frame-accurate**, and emits them into program as a **fill + key** pair. | Native GPU engines (Viz Engine, XPression, Chyron PRIME); web path = **Chromium/CEF offscreen** (CasparCG HTML producer, Singular, Videon at 60 fps). |

**Defining trait** (vs. offline motion graphics like After Effects): **real-time, frame-accurate, data-driven, command-triggered**, with transparency carried all the way to air.

## 2. What You Can Create — The Graphic Catalog

Everything the request asked about ("titles, sub-titles, overlays, animations, fly-ins, lists of people, upcoming lists…"), with the correct terminology and how each is built and animated.

### 2A. Titles & sub-titles
| Graphic | What it is | How it's built |
|---|---|---|
| **Full-screen / opening title** | Full-frame branded card (show open, bumper, segment title) | Heaviest animation — logo stings, kinetic type; often a baked **Lottie** (After Effects → JSON) sequence |
| **Lower third** | Graphic in the lower title-safe area identifying the current person/topic | Text over a plate/box/gradient scrim; transparency via alpha so video shows through. **Tiers:** 1-tier (name/label) · 2-tier (name + role/location = also a **locator**) · 3-tier (adds temporal context) |
| **Name super / name strap / aston / chyron / CG** | *All synonyms* for the person-ID lower third (name + title). Regional: "chyron"/"CG" (N. America), "name strap"/"aston" (UK) | A 2-tier lower-third template bound to `{name, title}` |
| **Byline / locator** | Reporter credit / place-date stamp | Lower-third variants |
| **Sub-titles vs captions** *(distinct from designed titles)* | **Subtitles** = translation of dialogue (assumes you can hear). **Captions** = dialogue + speaker IDs + sound description for deaf/HoH; **closed** = toggleable, **open/burned-in** = permanent | Rendered as a distinct caption *layer*, not a designed template; ties to the ANC/CC apps (AR.9.1) and the conditioner row's caption stages |

### 2B. Overlays / persistent graphics
| Graphic | What it is |
|---|---|
| **Bug / DOG (digital on-screen graphic) / logo** | Semi-transparent channel/brand watermark, persistent in a corner |
| **Ticker / crawl / zipper** | Thin horizontal bar of continuously scrolling data-bound text (news/scores/alerts) |
| **Breaking-news strap / stinger** | High-urgency full-width bar, animated in with an audio stinger, droppable live at any time |
| **Score bug / clock-and-score** | Persistent compact panel (teams, score, game clock, period) — live-updated **in place** |
| **Stat panel** | On-demand data card (player/team stats) |
| **Clock / countdown** | Time-of-day or event countdown driven by a running timer |

### 2C. Lists / data graphics — "upcoming lists" and "lists of people"
This is exactly what the request called out ("upcoming lists, lists of people to be there in the production"). All are built the same way: **a single row template bound to an array of records; the engine repeats the row per record** (`<template>` element + array binding).

| Graphic | Built from |
|---|---|
| **"Coming up next" / up-next / rundown list** | Array of upcoming segments → repeated row template |
| **Agenda / line-up list** | Array of items |
| **Participant / guest / talent list** | Array of people `{name, role, headshot}` → repeated row (directly the "people to be there") |
| **Roll-up end credits** | Long array translated vertically at constant speed (the "roll") |
| **Leaderboard / standings** | Array with live re-sort + position transitions |

**List animation:** **staggered entrance** (rows animate in sequentially, ~0.05–0.1 s offset), **roll/scroll** (credits/tickers translate at constant speed with seamless wrap), **update-in-place** re-sort (leaderboards).

## 3. Animation, Fly-Ins & Transitions

### 3A. The lifecycle every template follows
```
   ANIMATE-IN  →  HOLD / LOOP  →  ANIMATE-OUT
        (with UPDATE re-animating only changed data, no full out/in)
```
Exposed as **IN / UPDATE / OUT** states (CasparCG maps these to JS `play()` / `update()` / `next()` / `stop()`). The control system *triggers* these by command — the template does not auto-play.

### 3B. Named techniques
Fly-in / fly-out, slide, wipe, dissolve/crossfade, **reveal/mask wipe** (element revealed through an animating mask), typewriter (kinetic type), counter/odometer (number count-up), morph. Craft: easing (**ease-in-out** default, **overshoot/back** for snap), **anticipation** (small reverse before the move), **stagger**, follow-through — the classic animation principles applied to broadcast motion.

### 3C. Web implementation stack (since the renderer is HTML5)
| Tech | Use it for |
|---|---|
| **GSAP** (GreenSock) | The de-facto standard for broadcast web motion — **timelines** sequence many tweens, precise stagger, scrub/reverse/pause; ideal for the IN/HOLD/OUT model and for deterministic frame-stepping |
| **Lottie / Bodymovin** | Pre-baked After Effects vector sequences (logo stings, complex opens) shipped as JSON → rendered to SVG/canvas; tiny + infinitely scalable vs. GIF/PNG-sequence |
| **CSS keyframes / transitions** | Cheapest, GPU-accelerated for `transform`/`opacity`; respects `prefers-reduced-motion` |
| **Web Animations API (WAAPI)** | Native programmatic control bridging CSS and JS |
| **WebGL / WebGPU / shaders** | GPU wipes, displacement/reveal transitions, 3D, effects beyond DOM compositing |
| **Rive** | Interactive/stateful vector animation (integrated by some cloud broadcast tools) |

**60 fps rule:** the renderer must sustain the full channel frame rate (50/59.94/60) with **zero dropped frames** — far stricter than web UX. Animate only compositor-friendly properties (`transform`/`opacity`) to stay on the GPU; avoid layout thrash; keep DOM light; frame-step against a synthetic clock for drift-free output.

## 4. Alpha Layers & Compositing (how graphics get *over* video)

This is the part most people underestimate, and it's a hard requirement in AR.4.7.

- **Key + Fill** — the broadcast model. **Fill** = the RGB imagery; **Key** = the matte/alpha expressed as a black→white luma signal (black = transparent, white = opaque). A downstream **linear keyer / DSK** blends fill over program using the key. AR.4.7 mandates the renderer emit this pair — as `4:2:2 (fill) + 4:0:0 (key)` YUV, *or* RGB+Alpha.
- **Output transports:** on **SDI**, fill and key are two BNCs; on **ST 2110**, they are **two separate -20 video essences**, PTP-aligned (ST 2110-10). ANC/captions ride ST 2110-**40** alongside.
- **Straight vs. premultiplied alpha** — the classic source of "halo/fringe" bugs:
  - **Straight (unassociated):** RGB independent of alpha. Matches PNG/web semantics; cleanest to reason about.
  - **Premultiplied:** RGB already multiplied by alpha; the standard "over" equation `out = fg + (1−α)·bg` assumes it; avoids edge fringing on soft/antialiased edges.
  - **In the browser:** a **WebGL canvas is premultiplied by default** (opt out with `getContext('webgl',{premultipliedAlpha:false})`); Canvas 2D is premultiplied internally. When converting the RGBA framebuffer into fill+key, the pipeline **must know which** or you get dark/bright fringes. This is the #1 "alpha looks wrong" cause.
- **Z-order / stacking:** strict order — video base → full-frame → lists/panels → lower thirds → ticker → bug on top (CSS `z-index` within stacking contexts).
- **Masks/clips/blends:** CSS `mask`/`clip-path` (+ SVG) drive reveal wipes; `mix-blend-mode` for multiply/screen/overlay effects.
- **Title-safe / action-safe** (SMPTE ST 2046-1): **safe title = 90%** of frame (5% margins), **safe action = 93%** (3.5% margins). *All* text must sit inside title-safe so consumer overscan never clips it. Implement as a fixed global inset guide every template honors.
- **Legibility over live video:** never bare text — always drop shadows, outlines, or semi-opaque plates/gradient scrims behind text (feature-detect `filter: drop-shadow()` with a solid-plate fallback).

## 5. Data & "How Things Work" — The Runtime Flow

### 5A. Data binding
Templates expose typed named fields (text/number/image/color/data-source); a data layer maps live sources onto them. Sources: **JSON/REST, WebSocket** (low-latency), spreadsheets, databases, and sports/election/finance feeds.

**The pivotal distinction — update-in-place vs. re-take:**
- **Update-in-place:** the graphic stays on air and only its data mutates (score 2→3 with a subtle tween) via an `UPDATE` call. Essential for scorebugs/tickers so they don't re-animate constantly.
- **Re-take:** animate the old graphic out, the new one in. Typical for lower thirds.

### 5B. End-to-end, one graphic's life
```
1. Designer authors template  (HTML/CSS/JS + field schema + IN/UPDATE/OUT timeline)   [Layer 1]
2. Operator/journalist/automation fills fields with data                               [Layer 2]
3. Item placed in a rundown/playlist; staged on PREVIEW; verified                       [Layer 3]
4. TAKE → control sends "load URL + play" → renderer animates IN                        [Layer 3→4]
5. Data changes → UPDATE (in place) ; multi-state → NEXT                                [Layer 3→4]
6. Renderer emits FILL + KEY every frame at channel rate                                [Layer 4]
7. DSK keys fill over PROGRAM using the key                                             [downstream]
8. OUT → animate off ; REMOVE/CLEAR → cleanup                                           [Layer 3→4]
```

## 6. Control & Playout Model

- **Rundown / playlist:** an ordered list of graphic items with a highlighted "next" bar; operators cue/**take** live.
- **Preview vs. Program:** stage/verify on preview → **Take** to program. **Take/Next** advances states (e.g., reveal bullet points one at a time).
- **Control protocols (the API surface):**
  - **AMCP** (CasparCG's TCP text protocol): `CG ADD/PLAY/UPDATE/NEXT/STOP/INVOKE/REMOVE/CLEAR`, layers 1–999 for stacking. The cleanest lifecycle mapping to model against.
  - **REST + WebSocket** (cloud/HTML engines; public trigger URLs).
  - **OSC** (send-only status/telemetry from CasparCG).
  - **NRCS + MOS** for newsroom: the **NRCS** (iNews/ENPS/Octopus) owns the rundown; the **MOS protocol** (XML over TCP) links it to graphics/video/prompter devices via a **MOS Gateway**, with a template-chooser plugin embedded in the journalist's story editor. (Viz Flowics was the first HTML5 cloud graphics system to add an NRCS/MOS gateway.)
- **Presets & saved instances:** control apps store playlists, saved field values, template presets, brand kits.
- **Redundancy:** **1+1 main/backup** renderers driven by the *same* commands (mirrored AMCP/MOS), health monitoring to cut to backup; deterministic, command-driven templates let a hot standby hold identical on-air state.

### Presets & branding (what "presets" means here)
- **Style presets / template variants** — one design shipped in multiple looks (color/size/tier).
- **Theme / skin via design tokens** — colours, type, radius, shadow as **CSS custom properties**, swappable to reskin every template at once.
- **Brand kit** — central logos/fonts/palettes every template consumes from one source.
- **Saved graphic instances / playlists** — pre-filled, ordered, timed graphics ready for live recall.

## 7. How This Maps To TwistRouting (the delta)

**Current state:** there is **no graphics editor** in the codebase — only a `gfx` capability and a **"Tactical" role** ("Graphics & AR Lead — on-screen overlays, AR & data visualization"). So AR.4.7 and the whole engine are **L0** (the `gfx` cap is a scaffolded seam awaiting an editor). The Vision-Mixer editor *does* model **DSK (downstream keyer)** slots (L2) — the natural on-air destination for engine output — and the Signaling editor models triggers.

**The opportunity:** TwistRouting is HTML5-native, plugin-based, JSON-data-driven, and role-gated — an ideal home for the **engine's authoring + control + preview layers** (the out-of-scope-for-the-RFSA upstream tool). It should *drive* an AR.4.7 renderer, not re-implement the renderer.

| Engine layer | Build in TwistRouting as | Reuses |
|---|---|---|
| **Template model** | Templates as `Routes/**` JSON (field schema + IN/UPDATE/OUT metadata) — zero-backend discovery, like every other resource | discovery, plugin registry |
| **Data binding** | Field editor + live data preview; JSON/WebSocket data source stubs | source model, `routing-core` |
| **Control / playout** | A **"Graphics Engine" editor**: rundown/playlist strip, preview vs program, TAKE / NEXT / UPDATE / OUT buttons, per-item fields | `core.js` toolkit, overlay, role gating (`gfx`/`admin`) |
| **Preview render** | An in-GUI HTML5 preview of the template (the request's own GUI requirement: "URL + a preview of the rendered graphic") | overlay, iframe/canvas |
| **Output** | Represent the Fill+Key output as a twist onto the routing map → into a Vision-Mixer **DSK** | vision-mixer DSK, conditioner-row ANC/CC stages |

**Ties to prior work:** captions/CC handled by the conditioner row's caption stages (AR.9.1); tally/triggers by Signaling; the engine's output is one more conditioned signal on the routing spine; under **Interpretation B** (delta report), the same GUI serves as simulation today and real control (AMCP/REST/MOS) once wired.

**Out of scope here (by design):** the frame-accurate headless-Chromium rendering to fill/key essence is the **AR.4.7 renderer App's** job (a platform/vendor App). TwistRouting builds the **authoring, data, control, preview, and routing** around it.

## 9. Bottom Line

- **The request only specs a dumb renderer (AR.4.7); the "graphics engine" is the upstream authoring/data/control workflow it explicitly leaves out of scope** — that is the real system, and it is a four-layer architecture: **Design/Template → Data → Control/Playout → Render+Output (fill+key)**.
- **Everything asked for maps cleanly:** titles & name supers = tiered lower-third templates; overlays = bug/ticker/score/breaking; **"upcoming lists" and "lists of people" = a row template bound to an array**, animated with stagger + roll; fly-ins/transitions = the **IN/UPDATE/OUT** lifecycle driven by **GSAP** (+ Lottie for baked opens); alpha = **key+fill**, straight-vs-premultiplied handled, inside **title-safe (90%)**.
- **"How it works":** data binds into template fields; operators sequence via rundown/preview/program with **take/next/update/out** (AMCP / MOS / REST-WebSocket); the renderer emits fill+key every frame; a DSK keys it over program; **update-in-place** keeps live graphics on air without re-animating.
- **In TwistRouting:** a genuine **L0 gap today** (only a `gfx` capability + Tactical role). The high-value build is the **engine's authoring + control + preview surface** (a Graphics Engine editor + JSON template model), driving an AR.4.7 renderer and feeding the Vision-Mixer DSK — the same Interpretation-B pattern: real GUI now, real control when wired.

> **One sentence:** build the *engine* (templates + data + playout + preview), not the *renderer* — the renderer just paints your HTML to fill+key; the value is the system that authors the graphic, binds the data, sequences the show, and takes it to air.
